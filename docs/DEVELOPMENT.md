### Iteration 8：編譯相容性優化與 Windows 亂碼修復 (Week 8)

**目標**
- 解決 Windows 環境下 MSVC (Visual Studio) 編譯與執行時的中文亂碼問題
- 自動偵測並適應系統中已安裝的編譯器（MinGW g++ 或 MSVC cl.exe）

**任務**
- [x] 修復 `<Windows.h>` 的巨集衝突
  - 在 `src/main.cpp` 中引入 `<Windows.h>` 前加上 `#define NOMINMAX`，防止 Windows SDK 定義的 `min` 和 `max` 巨集干擾 C++ 標準庫的 `std::numeric_limits<streamsize>::max()`。
- [x] 自動偵測 Visual Studio 環境編譯
  - 升級 `build.bat`。如果找不到 `g++`，會自動調用 `vswhere.exe` 尋找本機的 Visual Studio 安裝路徑，載入 `VsDevCmd.bat` 的開發者命令環境，並調用 `cl.exe` 進行編譯。
- [x] 處理編譯源檔編碼
  - 在 MSVC 編譯指令中加入 `/utf-8` 參數，強制編譯器以 UTF-8 格式讀取源碼，避免中文字串常數在 Big5 語系系統中被截斷或造成語法錯誤。
- [x] 提供解決終端機亂碼的執行批次檔
  - 建立及更新 `START.bat` 與 `run_utf8_with_pause.bat`。利用 `chcp 65001` 指令切換 Windows 主控台字碼頁至 UTF-8，使輸出的中文選單能順利呈現，避免亂碼。

**輸出**
- 整合自動偵測編譯器的 `build.bat` 腳本
- 解決 `NOMINMAX` 巨集衝突的原始碼
- 跨編譯器相容且無亂碼的 `rental_system.exe` 執行檔

**技術亮點**
- ✅ 條件式編譯器排查與調用
- ✅ `/utf-8` MSVC 編譯開關
- ✅ 預處理器巨集衝突排解 (`NOMINMAX`)
- ✅ Windows 控制台字碼頁配置 (`chcp 65001`)

## 📚 技術決策記錄 (ADR)

### ADR-001：使用 shared_ptr 管理器材對象
**決策**：使用 std::shared_ptr 而不是原始指針
**原因**：
- 自動內存管理
- 避免內存洩漏
- 多態性支持

### ADR-002：CSV 文件格式存儲
**決策**：使用 CSV 格式而不是 JSON 或 XML
**原因**：
- 簡單易實現
- 適合當前需求
- 易於調試

### ADR-003：菜單驅動 UI
**決策**：使用菜單驅動終端 UI 而不是圖形界面
**原因**：
- 跨平臺兼容性
- 易於實現
- 教育用途明確
